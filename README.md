# Character Insight Canvas

一个用于深度理解虚构角色，并将研究结果组织为“可追溯、可探索、只读单文件 HTML 画板”的通用 Agent Skill。

它不限定作品媒介。游戏、动画、漫画、小说、电影、舞台剧、广播剧以及跨改编角色都可以使用。

## 它解决什么问题

普通人物资料往往有两个极端：

- 只有百科信息，知道角色设定，却不知道角色为什么鲜明；
- 只有主观分析，观点很多，却无法回到剧情和来源核对。

这个 Skill 使用“角色优先、证据下钻”的结构：

```text
角色速览 → 核心路径 → 证据地图 → 节点与出处
```

读者先理解角色是谁、第一印象如何变化、性格怎样通过行动体现，再进入关系、经历、机制、阵营、主题或改编差异等证据层。

## 核心能力

- 从用户材料或公开来源建立论断台账；
- 区分文本事实、官方补充、二手重建、解释和视觉观察；
- 提炼“第一眼 → 深入后”的认知变化；
- 用行为和选择描述性格，不堆砌形容词；
- 建立关键关系张力和角色变化路径；
- 根据实际问题生成最少且必要的证据地图；
- 优先使用官方或作品内视觉参考；
- 生成或更新可离线打开的单文件 HTML 角色探索画板；
- 对数据引用、抽屉导航、移动端、控制台和离线请求执行验收。

## 地图数量不是模板

本 Skill 不规定“两张”“三张”或“四张”地图。

地图数量由材料需要回答的独立问题决定：

- 简单角色可能只有人物关系图；
- 存在明显成长顺序时，可以增加主观时间或角色弧地图；
- 涉及能力、制度或世界规则时，可以增加机制地图；
- 不同媒介改编存在冲突时，可以增加版本差异地图；
- 两张地图回答同一个问题时，应当合并。

原则只有一个：使用能够回答问题的最小地图集合。每张地图都必须证明自己有存在价值。

## 仓库结构

```text
.
├── README.md
└── skills/
    └── character-insight-canvas/
        ├── SKILL.md
        ├── references/
        │   ├── content-contract.md
        │   ├── quality-gates.md
        │   ├── renderer-contract.md
        │   └── research-contract.md
        └── scripts/
            ├── audit-package.mjs
            └── scaffold.mjs
```

其中：

- `SKILL.md`：执行入口和七个完成门槛；
- `research-contract.md`：来源等级、论断类型和冲突处理；
- `content-contract.md`：角色速览、核心路径、证据地图和详情层结构；
- `renderer-contract.md`：通用只读画板与单文件构建要求；
- `quality-gates.md`：研究、信息架构、技术和浏览器验收清单；
- `scaffold.mjs`：生成新角色数据脚手架；
- `audit-package.mjs`：检查 Skill 包结构、frontmatter、私有路径和固定地图数量。

## 安装

### Codex 用户级安装

克隆仓库后，将 Skill 目录复制到 Codex 用户技能目录：

```bash
mkdir -p ~/.codex/skills
cp -R skills/character-insight-canvas ~/.codex/skills/
```

### 项目级、跨 Agent 安装

对于支持 Agent Skills 的项目，推荐放入项目的 `.agents/skills/`：

```bash
mkdir -p .agents/skills
cp -R skills/character-insight-canvas .agents/skills/
```

不同 Agent 的发现路径、权限和调用界面并不完全相同。本仓库的共享 `SKILL.md` 只使用保守的 `name` 与 `description` frontmatter；当前工作流已在 Codex 环境中验证，其他 Agent 请按各自版本重新验证。

新安装的 Skill 如果没有立即出现在当前会话中，请新建会话，让 Agent 重新扫描技能目录。

## 使用

在支持显式 Skill 调用的环境中：

```text
$character-insight-canvas 深度研究《作品名》的角色名，并生成单文件角色探索画板
```

也可以自然描述任务：

```text
基于动画第一季研究这个角色，不要使用漫画剧透，并生成角色探索 HTML。
```

```text
比较这个角色在漫画与动画改编中的差异，把冲突证据单独做成地图。
```

```text
更新现有角色画板，补充第一印象、鲜明特点、视觉参考和来源标注。
```

当角色同名、改编边界不清或剧透范围会改变产出时，Skill 才应提出阻塞性问题；其他情况默认直接研究和执行。

## 新建角色数据

需要创建新案例时，可以先运行脚手架：

```bash
node skills/character-insight-canvas/scripts/scaffold.mjs \
  --character "角色名" \
  --work "作品名" \
  --output "data/character.canvas.json"
```

脚手架故意包含 `RESEARCH REQUIRED` 占位符。它只能作为数据起点，不能直接当成完成品。最终构建前必须完成研究、替换全部占位符并通过数据验证。

## 资料与证据规则

来源优先级默认如下：

1. 作品正文、剧集、章节、路线、正式脚本或用户提供的原始材料；
2. 官方人物页、制作资料、作者说明和正式访谈；
3. 可靠的场景记录、授权资料库或详细重建；
4. 评论、分析、Wiki 和讨论资料；
5. 同人图、Cosplay、编辑图和情绪板。

第五级只能作为“再诠释参考”，不能冒充官方视觉设定。资料发生冲突时，保留冲突和来源范围，不得把多个版本平均成虚假的确定答案。

## HTML 与渲染器边界

当前仓库提供的是 Skill、研究契约、数据脚手架和渲染要求，不包含一套独立发布的 WebGL 渲染器实现。

Skill 会优先复用工作区内满足 `renderer-contract.md` 的兼容渲染器。如果没有兼容渲染器，而用户明确要求 HTML 产物，Agent 需要实现满足契约的最小渲染器；不能用静态截图或假交互页面冒充完成品。

合格的最终 HTML 应当：

- 内联 CSS、JavaScript、角色数据和必要图片；
- 不依赖外部脚本、样式表或初始 `fetch()`；
- 本地双击即可打开；
- 默认先展示角色速览；
- 支持核心路径、证据地图、节点详情和折叠抽屉；
- 在桌面与移动视口下可读；
- 控制台无错误，初始网络请求只有 HTML 本身。

## 自检

运行 Skill 包审计：

```bash
node skills/character-insight-canvas/scripts/audit-package.mjs
```

当前审计会检查：

- 必需文件是否存在；
- frontmatter 是否只有 `name` 和 `description`；
- Skill 名称与触发描述是否合法；
- 是否残留本机绝对路径；
- 是否把证据地图数量写成固定区间。

## 有意不包含的内容

- 不内置任何具体角色资料；
- 不绑定游戏路线、动画集数或漫画章节；
- 不保证公开网络资料天然正确；
- 不把同人视觉当作官方证据；
- 不声称不同 Agent 具有完全相同的执行效果；
- 不包含许可证文件。
